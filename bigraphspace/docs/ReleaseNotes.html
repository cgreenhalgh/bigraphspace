Release Notes - ReactiveXML/Bigraphs

Chris Greenhalgh

Known issues

  • The abstraction for Match is massively incomplete, and there is no abstract access to Matcher/Transformer (c.f. Bigraph vs DomBigraph)
  • Matcher (and Bigraph/Place) does not distinguish between support names and support variable names - needed to match to allow matching to specific individuals.
  • There is currently no support in the Definitions XML or the code to specify that a rule applies only if the context of the match is (transitively) active. 
  • BigraphFinder has only a single  global default - should allow (e.g. class-loader specific default for use in simulator)
  • DomReactiveBigraph has some placeholder fields but doesn't actually support creation as a sub-bigraph.
  • RuleCondition evaluation does not do active test.
  • Control index matching in the DomMatcher is by string equality, so will not be correct if string representation is not normalised for type.
  • DomMatch variable values are all strings.
  • Not all bigraph variable methods are currently implemented (in the programmatic manipulation parts of the API).
  • (related) The VariableDefinition and VariableConstraint classes should become interfaces with DomBigraph and signature.xml/standalone implementations.
  • Control and support-type bigraph variables are not currently implemented/supported. 

Possible enhancements

  • Some ReactionRule conditions would be easier to specify if multiple conditions/redex could share link value binding constraints, i.e. value bindings. For example it would be easier to say (in effect) provided that this particular X is not already known... (perhaps preconditions after the redex using that match as a basis)
